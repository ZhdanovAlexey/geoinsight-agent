# GeoInsight Agent — видение продукта

> **Версия:** 2 (2026-04-18)
> **Предыдущая:** v1 от 2026-04-17 (`2026-04-17-geoinsight-agent-design.md`)
> **Изменения относительно v1:** см. раздел [Changelog](#changelog) в конце.

## Контекст

Мобильные операторы накапливают огромные объёмы геоданных: где находятся абоненты, когда, как перемещаются между домом и работой, какой у них демографический профиль. Эти данные — ценный актив для бизнеса, но доступ к ним требует экспертизы в GIS, SQL и аналитике.

**GeoInsight Agent** — AI-ассистент, который превращает телеком-геоданные в бизнес-аналитику через чат-интерфейс на естественном языке. Клиент задаёт бизнес-вопрос — получает ответ с картами, таблицами и рекомендациями.

## Целевая аудитория

B2B-клиенты мобильного оператора:
- Ритейл и HoReCa
- Недвижимость и девелопмент
- Реклама и маркетинг
- Транспорт и логистика
- Туризм

## Данные

Агент работает с геоданными оператора, которые включают:

| Тип данных | Описание | Пример из dataset |
|---|---|---|
| **Справочник зон** | Полигоны ~250x250м с привязкой к городу | `dim_zid_town_Olmaliq.csv`: zid → WKT polygon |
| **Демография зон** | Доход, возраст, пол, количество людей — агрегат | `geo_olmaliq_cnt.csv`: zid × income × age × gender → cnt, home_zid, job_zid |
| **Динамика зон** | То же, но с временной осью — как меняется население зоны во времени | `geo_olmaliq_dyn_all.csv`: zid × timestamp × income × age × gender → cnt |
| **Траектории** | Индивидуальные перемещения абонентов: дом, работа, почасовая локация | `tav_geo_tosh.csv`: code × age × home/job_zid × hourly_zid × roaming |
| **Роуминг** | Тип сети (домашняя / внутрисетевая / международная), страна | Часть траекторий: roaming_type, country_name |

## Юзкейсы по вертикалям

> **Покрытие в MVP:** ✅ полностью / 🟡 stub-тул возвращает заглушку / ❌ вне scope MVP. См. раздел [MVP scope](#mvp-scope).

### 1. Ритейл и HoReCa

**Site Selection — выбор локации для новой точки** ✅
- "Где лучше открыть кофейню для аудитории 25-35 с доходом выше среднего?"
- Агент ищет зоны с высокой концентрацией целевой демографии, анализирует проходимость по времени суток, строит карту с рейтингом зон

**Catchment Analysis — анализ зоны охвата** 🟡
- "Сколько людей живёт и работает в радиусе 1 км от моего магазина?"
- Агент определяет зоны в радиусе, суммирует population, строит демографический профиль

**Конкурентный анализ (при наличии данных о точках)** ❌
- "Какая проходимость у зоны конкурента на ул. Навои vs у нас?"
- Сравнение зон по трафику, демографии, пиковым часам

**Оценка каннибализации** ❌
- "Если я открою вторую точку в 2 км от первой, сколько аудитории пересечётся?"
- Анализ пересечения catchment areas через потоки дом-работа

### 2. Недвижимость и девелопмент

**Оценка привлекательности района** ✅
- "Кто живёт в этом районе? Какой средний доход, возраст, сколько людей?"
- Демографический профиль зон + динамика (растёт/падает население)

**Маятниковая миграция** 🟡
- "Откуда приезжают люди, работающие в центре? Где они живут?"
- Анализ home_zid для людей с job_zid в заданной зоне → карта потоков

**Спрос на инфраструктуру** 🟡
- "В каких районах больше всего людей, но мало рабочих мест?"
- Соотношение home vs job count по зонам → выявление "спальных" и "деловых" районов

### 3. Реклама и маркетинг

**Оценка OOH-локаций (наружная реклама)** ✅
- "Сколько людей 18-35 проходит мимо этой точки в день? В какое время пик?"
- zone_traffic с фильтром по демографии + почасовая динамика

**Геотаргетинг** ✅
- "Найди топ-20 зон с максимальной концентрацией женщин 25-35 с доходом выше среднего"
- find_zones с мультипараметрическим фильтром → карта + таблица

**Планирование рекламных кампаний** ✅
- "В какое время суток в деловом центре максимум людей 30-45?"
- Почасовая динамика с демографическим фильтром

### 4. Транспорт и логистика

**Карта потоков** 🟡
- "Покажи главные маршруты маятниковой миграции в городе"
- Агрегация home→job потоков, визуализация на карте линиями разной толщины

**Размещение пунктов выдачи / складов** ❌
- "Где разместить 3 пункта выдачи, чтобы покрыть максимум населения?"
- Кластеризация + оптимизация покрытия

**Пиковые нагрузки** 🟡
- "Какие зоны перегружены в 8-9 утра? Откуда и куда едут люди?"
- Анализ временных интервалов из траекторий

### 5. Туризм

**Анализ международных визитов** 🟡
- "Сколько иностранцев было в городе? Из каких стран? Куда ходят?"
- Фильтр по roaming_type = "Международный", группировка по country_name + популярные zid'ы

**Маршруты туристов** 🟡
- "Какие зоны посещают казахстанские туристы? В какое время?"
- Анализ почасовых перемещений для абонентов с country_name = "Казахстан"

## Технологический стек

| Слой | Технология | Обоснование |
|---|---|---|
| LLM | `openai/gpt-oss-120b` через vLLM | Open-weight (Apache 2.0), сильный в reasoning и tool use, локальный хостинг — критично для телеком-данных |
| Агентный фреймворк | OpenAI Agents SDK (Python) | Официально поддерживает связку с vLLM, нативная интеграция с OpenTelemetry |
| Backend | FastAPI + Uvicorn | Async, OpenAPI из коробки, удобный SSE |
| API контракт | OpenAI-compatible Chat Completions + SSE | Любой существующий OpenAI-клиент работает; смена UI не требует менять бэкенд |
| БД | PostgreSQL + PostGIS | Спатиальные индексы, GeoJSON-сериализация в одну функцию, зрелая экосистема |
| Demo UI | Streamlit + pydeck | Быстрый прототип; pydeck (deck.gl) переносим в прод-фронт без переписывания |
| Логирование | structlog (JSON) | Structured-логи с trace_id для корреляции |
| Observability | Langfuse self-hosted | LLM-specific трейсинг, on-premise (важно для приватности), нативная интеграция с Agents SDK через OTEL |
| Деплой (демо) | Docker Compose | Локальный запуск всего стека одной командой |

## Контракт API

### Endpoint

`POST /v1/chat/completions` — совместим с OpenAI Chat Completions API.

**Request** (стандартный):
```json
{
  "model": "geoinsight-v1",
  "messages": [
    {"role": "user", "content": "Где открыть кофейню для аудитории 25-35?"}
  ],
  "stream": true
}
```

**Response при `stream: false`** — стандартный `ChatCompletion` с дополнительным полем `artifacts` на верхнем уровне (стандартный OpenAI-клиент его проигнорирует — это допустимое расширение):

```json
{
  "id": "chatcmpl-abc123",
  "object": "chat.completion",
  "created": 1745000000,
  "model": "geoinsight-v1",
  "choices": [{...стандартное...}],
  "usage": {...},
  "trace_id": "lf-uuid",
  "artifacts": [
    {"id": "art_1", "type": "map", ...},
    {"id": "art_2", "type": "table", ...}
  ]
}
```

**Response при `stream: true`** — SSE-стрим. Стандартные чанки модели идут как `data: {...}`. Кастомные события идут с префиксом `event:`:

```
data: {"choices":[{"delta":{"role":"assistant"}}]}

event: trace_started
data: {"trace_id":"...","langfuse_url":"..."}

event: tool_started
data: {"call_id":"call_1","name":"find_zones","args":{...}}

event: tool_finished
data: {"call_id":"call_1","duration_ms":2340,"rows":127}

event: artifact
data: {"id":"art_1","type":"map","geojson":{...}}

data: {"choices":[{"delta":{"content":"Нашёл "}}]}
data: {"choices":[{"delta":{"content":"12 зон..."}}]}
data: [DONE]
```

**Перечень кастомных SSE-событий:**

| event | payload | назначение |
|---|---|---|
| `trace_started` | `trace_id`, `langfuse_url?` | связь с observability в начале запроса |
| `tool_started` | `call_id`, `name`, `args` | UI показывает «выполняю инструмент» |
| `tool_progress` | `call_id`, `progress`, `message?` | опционально для долгих тулов |
| `tool_finished` | `call_id`, `duration_ms`, `status`, `rows?` | UI показывает завершение |
| `tool_failed` | `call_id`, `duration_ms`, `error`, `error_type` | для отображения ошибки |
| `artifact` | объект Artifact (см. ниже) | передача карты/таблицы/графика для рендера |

## UI-артефакты — контракт между бэкендом и фронтом

LLM работает с **компактными summary** в tool outputs. Полные геометрии и таблицы передаются отдельно через артефакты — чтобы:
- не жечь контекст модели полигонами на 200 зон
- фронт получал данные в готовом для рендера формате
- одни и те же данные могли использоваться разными UI (Streamlit demo сейчас, прод-фронт потом)

**Типы артефактов:**

| type | назначение | ключевые поля |
|---|---|---|
| `map` | choropleth / heatmap / точечная карта | `viz`, `geojson`, `color_metric`, `legend`, `bbox`, `tooltip_fields` |
| `flow_map` | потоки (например, маятниковая миграция) | `flows: [{from, to, weight}]`, `bbox` |
| `table` | табличные данные | `title`, `columns`, `rows` |
| `chart` | график (bar/line/pie) | `chart_type`, `title`, `data` |

**Принципы:**
- Каждый тул, генерирующий визуальные данные, **сам эмитит свой артефакт** — отдельного `render_map`-тула нет (антипаттерн: зря тратит токены модели).
- Геометрия отдаётся как **GeoJSON**, не WKT — фронт-библиотеки едят нативно.
- Артефакт должен быть самодостаточным — фронт не должен дёргать бэкенд для дорендера.

## Описание Tools

> Все тулы:
> - возвращают **компактный summary** для LLM (без полных полигонов и без полных таблиц >10 строк);
> - вызывают `ctx.emit_artifact(...)` для передачи UI-артефакта;
> - логируют события через structlog с `trace_id`;
> - попадают в Langfuse как отдельные span'ы.

### zone_demographics
- **Вход**: zid или список zid'ов, опционально фильтры (income, age, gender)
- **Выход для модели**: распределение по доходу, возрасту, полу; total count; count по home/job
- **Артефакт**: `table` с распределением (опционально `chart`)
- **Источник**: geo_olmaliq_cnt

### zone_traffic
- **Вход**: zid, временной диапазон (опционально), фильтры демографии
- **Выход для модели**: почасовые/временные агрегаты (peak_hour, peak_value, by_hour summary)
- **Артефакт**: `chart` (bar по часам)
- **Источник**: geo_olmaliq_dyn_all

### home_work_flow
- **Вход**: zid (или список), направление (from_home / to_work / both)
- **Выход для модели**: топ-N зон, откуда приходят / куда уходят люди; объём потока
- **Артефакт**: `flow_map` (arcs)
- **Источник**: geo_olmaliq_cnt (home_zid, job_zid), trajectories

### find_zones
- **Вход**: критерии поиска — демография, минимальная проходимость, координаты центра + радиус, город
- **Выход для модели**: top-N зон со score и краткими метриками (без геометрии)
- **Артефакт**: `map` (choropleth по score) + `table`
- **Источник**: zones + zone_demographics + zone_dynamics

### compare_zones
- **Вход**: список zid'ов (2-5), набор метрик для сравнения
- **Выход для модели**: компактная таблица сравнения
- **Артефакт**: `table` + опционально `map` со сравниваемыми зонами
- **Источник**: все таблицы

### roaming_analysis
- **Вход**: город/регион, тип роуминга (международный / внутрисетевой), страна (опционально)
- **Выход для модели**: количество уникальных посетителей, топ-зоны посещения, временные паттерны
- **Артефакт**: `map` (heatmap по визитам) + опционально `chart` (динамика по времени)
- **Источник**: trajectories (roaming_type, country_name)

### catchment_area
- **Вход**: координаты точки или zid, радиус (метры), фильтры демографии
- **Выход для модели**: список зон в радиусе, суммарный профиль аудитории
- **Артефакт**: `map` с подсветкой зон в радиусе + `table` с профилем
- **Источник**: zones (геометрия) + zone_demographics

> ⚠️ Удалён тул `render_map` из v1 — визуализация теперь интегрирована в каждый тул через `emit_artifact()`. Это экономит токены и упрощает оркестрацию.

## Observability и логирование

### Langfuse (LLM-трейсинг)

**Self-hosted** (Docker Compose) — критично для приватности геоданных оператора.

Каждый HTTP-запрос = 1 trace в Langfuse, внутри:
- `agent.run` — root span
- `llm.completion` — каждый вызов vLLM (input/output, токены, **reasoning_content** от gpt-oss harmony как metadata)
- `tool.{name}` — каждый вызов тула с args и output

В response клиента (`trace_id` в JSON и `event: trace_started` в SSE) возвращается ссылка на trace в Langfuse — для дебага во время демо одним кликом.

### Structured logs (structlog)

JSON-логи в stdout/файл со связью через `trace_id`:

| Event | Когда |
|---|---|
| `request.received` | вход в `/v1/chat/completions` |
| `agent.started` / `agent.finished` | границы run |
| `tool.started` / `tool.finished` / `tool.failed` | каждый вызов тула |
| `artifact.emitted` | каждый эмит UI-артефакта |
| `request.failed` | глобальный exception handler |

### UI-индикация

Streamlit (и любой фронт, парсящий кастомные SSE-events) показывает:
- В реальном времени: какой тул сейчас выполняется, сколько занял, сколько строк вернул
- При ошибке: `tool_failed` с понятным сообщением
- В конце: ссылку на полный trace в Langfuse

Это решает две задачи одновременно: пользователь видит, что система работает (а не зависла), а разработчик имеет полную картину для отладки.

Агент: Нашёл 12 зон, подходящих под ваши критерии. Вот топ-5:

| # | Зона    | Людей/день | 18-35 лет | Доход 2+ | Рейтинг |

## MVP scope

### Входит в MVP

**Инфраструктура:**
- ✅ FastAPI бэкенд с OpenAI-compatible `/v1/chat/completions` (stream + non-stream)
- ✅ Streamlit demo-UI с прогрессом вызовов тулов и pydeck-картами
- ✅ PostgreSQL+PostGIS, ETL для одного-двух городов (Olmaliq, Tashkent)
- ✅ Langfuse self-hosted с полными трейсами + structlog с trace_id
- ✅ Docker Compose для локального запуска (PG + Langfuse)

**Тулы (полная реализация):**
- ✅ `find_zones`
- ✅ `zone_demographics`
- ✅ `zone_traffic`

**Тулы (stub — возвращают `{"status": "not_implemented"}`):**
- 🟡 `home_work_flow`, `catchment_area`, `compare_zones`, `roaming_analysis`

Stub нужны, чтобы system prompt описывал полный набор возможностей и модель могла планировать многошаговые сценарии — позже эти тулы заполнят данными по мере готовности.

### НЕ входит в MVP

- ❌ Аутентификация / API keys на бэкенде
- ❌ Multi-tenant
- ❌ Rate limiting и квоты
- ❌ Кеширование ответов модели
- ❌ Геокодинг адресов (address → zid)
- ❌ Поддержка городов кроме Olmaliq и Tashkent
- ❌ Деплой в облако / Kubernetes
- ❌ Real-time данные (только агрегаты с задержкой)
- ❌ Конкурентный анализ и оценка каннибализации (нет данных о точках клиентов)

## Масштабирование

Текущий dataset — демо на Олмалике и Ташкенте (Узбекистан). В продакшене:
- Данные будут по России в аналогичном формате
- Масштаб: тысячи городов, миллионы зон, десятки миллионов траекторий
- Data Layer должен поддерживать шардирование по регионам (партиционирование PG по `city`/`region`)
- Tools должны работать с параметром `region`/`city` для ограничения scope запросов
- vLLM можно горизонтально масштабировать за load balancer'ом — Agents SDK поддерживает любой OpenAI-compatible endpoint
- Langfuse self-hosted на отдельной инфре, обработка асинхронная (BatchSpanProcessor) — не влияет на латентность инференса

## Ограничения и риски

- **Приватность**: все данные агрегированы или анонимизированы (код абонента ≠ идентификация). В логах и трейсах **никогда не должны появляться** идентификаторы абонентов (`code` из траекторий) — маскирование на уровне тулов до возврата результата.
- **Актуальность**: данные фиксируются с задержкой, real-time не поддерживается
- **Покрытие**: только абоненты данного оператора (не 100% населения)
- **Геокодинг**: адрес → zid требует дополнительного слоя (Nominatim / Яндекс Геокодер) — вне MVP
- **Reasoning content модели**: gpt-oss harmony генерирует chain-of-thought. CoT логируется в Langfuse для дебага, но **не отдаётся** клиенту в `/v1/chat/completions` — может содержать промежуточные размышления, не пригодные для пользователя.
- **Галлюцинации zid**: system prompt должен явно запрещать модели "выдумывать" zid'ы — только те, что вернулись из тулов. На стороне UI желательно валидировать zid из тула против БД перед рендером.
- **Стоимость инференса**: gpt-oss-120b на vLLM — это GPU-инфра (минимум A100/H100). При расчёте unit-economics учитывать стоимость держания GPU 24/7.
